
# Week 3 Reflection: Collaboration, Standards, and Building on What Exists

This week felt different.

The focus shifted from working individually to working within a shared system — contributing to something that others had already started and helping move it forward.

Here are the key lessons and moments that stood out.

---

## 1. From Waiting to Contributing

At the start of the week, I was in a bit of an in-between space.

I had already been assigned and was working on **Phase 2 (Issue #123)**, but I was still waiting to be assigned something specific for **Phase 2A (fixing Ramalama documentation)**.

At first, this felt like a pause.

But then I remembered something important:

Open source is not about waiting to be told what to do.  
It is about building on what already exists.


So instead of waiting, I:

* reviewed existing proposals
* went through the HackMD collaboration document
* identified where I could meaningfully contribute

That shift changed how I approached the entire week.

---

## 2. Contributing to the Ramalama Documentation Style Guide

Most of my work this week focused on Phase 2A — improving the Ramalama documentation.

This was not just about writing content, but about:

* aligning on structure
* improving readability
* creating consistency across contributors

I built on the work that had already been started and contributed in two key ways.

First, I introduced a **usability and user experience section** to the style guide.

The existing guide was strong in structure and formatting, but I wanted to extend it by focusing on how users actually interact with the documentation.

This included:

* time to first success
* quick start examples
* progressive learning (from basic to advanced)
* reducing cognitive load

The goal was simple:
make the documentation easier to understand and use, especially for first-time users.

---

## 3. Using Vale to Strengthen Documentation Quality

I also ran the style guide through **Vale**, a tool used in Fedora and Red Hat documentation.

Vale functions like a grammar and style checker, but aligned with strict technical writing standards.

This helped identify:

* inconsistent wording
* formatting issues
* readability improvements
* style guide violations

Instead of leaving this as raw output, I organized the feedback into a **structured cleanup plan**, grouped into sections such as:

* terminology consistency
* heading structure
* readability
* passive voice
* formatting

This made it easier for contributors to collaborate and work through improvements together.

---

## 4. Taking Initiative Through Peer Review

One of the most impactful things I did this week was **spearheading a peer review exercise**.

Since mentor feedback was delayed due to a work trip, I created a structured way for us to support each other.

I:

* helped structure the HackMD page to support collaboration
* reviewed the work of three peers
* received feedback on my own work

This turned out to be extremely valuable.

Even without immediate mentor input, we were able to:

* improve our work
* catch issues early
* learn from each other’s approaches

It reinforced that collaboration is not something that happens after feedback — it is something we actively create.

---

## 5. Collaboration Over Perfection

This phase made one thing very clear:

The goal is not to have the “best idea.”  
The goal is to help the team move toward a shared standard.

I focused on:

- contributing ideas clearly
- building on existing proposals
- responding to others’ suggestions
- leaving visible, traceable input

This made my contributions more aligned and more useful to the group.

---

## 6. Iterating My Previous Work

Alongside Phase 2A, I continued refining my earlier work.

I:

* updated my Docling repository based on peer review feedback
* improved structure and clarity in my README
* strengthened my portfolio

Because I had completed the more complex parts earlier, I now had the space to revisit and improve them properly.

---

## 7. Becoming More Comfortable with Fedora

I also spent more time working directly within the Fedora environment this week.

What initially felt unfamiliar is starting to feel more natural.

* navigating the system is easier
* running pipelines is more intuitive
* troubleshooting feels more manageable

This has made the entire workflow smoother and more enjoyable.

---

## 8. The Value of Doing the Hard Things First

My biggest takeaway this week remains the same:

Do the hard things first.

Earlier in the process, I focused on the more challenging parts of the project.

At the time, it felt difficult.

But now, that is what has allowed me to:

* iterate on my work
* improve based on feedback
* contribute more meaningfully in Phase 2

If I had delayed those parts, I would not have had this flexibility.

---

## Final Thoughts

This week was less about producing new work and more about refining, aligning, and collaborating.

* contributing without waiting for assignment
* improving documentation quality
* working with others to build shared standards
* supporting each other through peer review

More than anything, it reinforced that open source is not an individual effort.
It is a collective process.
